## What this is

**Reps** — a strength-training workout logger built in **React Native + JavaScript on Expo**. It is being ported from an original SwiftUI concept. This app directory lives inside the `Reps` repo (at `project/app/`); the repo originated from a designer-skills toolkit, and the repo-root `CLAUDE.md` two levels up documents those design *skills*, not this codebase.

The **source of truth for the UI is screenshots**, not Figma, in `../brief/screenshots/` — a design-system sheet (colors + SF Pro type scale) plus a 9-screen flow. Note: those PNG filenames contain a narrow no-break space (` `) before "PM" — glob for them, don't hardcode the path. Project status, decisions, and run notes are tracked in `../STATE.md` — read it first; it is kept current.

## Expo version

Pinned to **Expo SDK 54** (RN 0.81, React 19.1) — do NOT bump it or `expo install --fix` toward a newer SDK. The physical iPhone runs via App Store Expo Go, which only supports SDK 54; a newer SDK breaks the phone (the App Store Expo Go lags npm). Read the versioned docs at https://docs.expo.dev/versions/v54.0.0/ before writing Expo-specific code. (`AGENTS.md` carries this same note.)

## Running the app

Uses a **development build** (`expo-dev-client`), NOT Expo Go, on simulator/Android — this decouples the app from whatever Expo Go SDK is installed. The native `ios/` and `android/` dirs are git-ignored and generated by prebuild (`npx expo prebuild` regenerates them).

- **Day-to-day** (dev app already installed): `npx expo start --dev-client`, then open the installed **Reps** app on the sim/emulator (JS hot-reloads from Metro).
- **First/native rebuild — iOS simulator:** `npx expo run:ios`
- **First/native rebuild — Android:** `./scripts/run-android.sh` — sets `JAVA_HOME` to Android Studio's bundled JBR (there is no `java` on PATH) and `ANDROID_HOME`, boots the `Pixel_7` AVD if no device is attached, then runs `expo run:android`.
- **Physical iPhone:** `npx expo start` and scan the QR with Expo Go (works because the project is SDK 54).

## Verifying changes

There is **no test runner or linter configured** (the `npm test`/lint scripts do not exist). Verify by:
- **Bundle check without a device:** `npx expo export --platform ios --output-dir /tmp/check` — catches import/syntax errors fast.
- **Project health:** `npx expo-doctor`.
- **On-device:** screenshot the running app — `xcrun simctl io booted screenshot out.png` (iOS) or `adb exec-out screencap -p > out.png` (Android).

## Architecture

**Design tokens are a strict two-layer system — this is the central convention.**
- `src/theme/primitives.js` — raw values transcribed 1:1 from the screenshot (color ramps by number, the SF Pro type scale). Never reference these from screens.
- `src/theme/theme.js` — *semantic* tokens (`colors.bg`, `colors.accent`, `colors.textPrimary`, `typography.h1`, `spacing`, `radius`) that map primitives to meaning. **Screens import only from `src/theme`** (the `index.js` barrel) and must use semantic tokens — no hardcoded hex/sizes. Brand accent is orange `#F05305`; the app is dark-only; `fontFamily` is left as the platform default because SF Pro is the iOS system font. Spacing/radius are an inferred 4-pt system (not measured from the screenshot) — confirm against real screens.

**Navigation** (`src/navigation/RootNavigator.js`) is the app shell: a root native-stack wraps a bottom-tab navigator. Tabs = Home / History / Profile. Pushed over the tabs: `PlanDetail`, `WorkoutDetail`, `InSession` (presented as `fullScreenModal` — the core logging flow), `Settings`, `Calendar`. The React Navigation `DarkTheme` is overridden with the semantic color tokens so navigation chrome matches. `App.js` mounts this inside a `SafeAreaProvider`.

**Screens** (`src/screens/`) — one file per screen. Several are still `ScreenPlaceholder` stubs (`src/components/ScreenPlaceholder.js`) that only wire up navigation; `HomeScreen` and `InSessionScreen` are the built-out ones. Replace a placeholder's body with real UI as each screen is built.

**Data is mock and deliberately isolated.** `src/data/mock.js` (home/library) and `src/data/session.js` (active workout) return plain shapes that screens read. **Persistence will be Supabase** (`@supabase/supabase-js` + auth) — keep the screen-facing data shapes stable so the backend swaps in behind them without rewriting views. Product decisions already locked (see `../STATE.md`): kg/lb unit toggle (default kg), user-built routines plus optional pre-built ones, Supabase backend.

**Shared components** live in `src/components/` (e.g. `AppButton` with `primary`/`secondary` variants) — reuse these rather than restyling buttons per screen.
