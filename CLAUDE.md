### Reps in one screen
- **Stack:** React Native + JavaScript on **Expo SDK 54** (RN 0.81, React 19.1), dark-only, iOS-first. **Do NOT bump the SDK or `expo install --fix` toward a newer one** — the user's physical iPhone runs App Store Expo Go, which only supports SDK 54 (see [project/app/AGENTS.md](project/app/AGENTS.md)). Read the versioned docs at https://docs.expo.dev/versions/v54.0.0/ before writing Expo code.
- **Run — two modes (runtime UI tiers):** the app picks `liquid-glass | material-you | fallback` per launch (`src/theme/useUITier.ts`). **Expo Go** (`cd project/app && npx expo start --go`, scan the QR / press `i`/`a`) always shows the **fallback** tier — fast, no build, works on the SDK-54 iPhone. A **dev build** (`npx expo run:ios` for the sim, or `npx expo run:ios --device` for the phone — needs Apple ID signing + USB-C) shows **liquid-glass** on iOS 26 and **material-you** on Android 12+. `expo-dev-client` is installed, so plain `npx expo start` now defaults to **dev-build** mode — use `--go` for Expo Go. Liquid Glass compiles only under **Xcode 26 / iOS 26 SDK**. `ios/`+`android/` are git-ignored, regenerated by prebuild / `expo run:*`.
- **No test runner or linter exists** (the `npm test`/lint scripts are absent). Verify with `npx expo export --platform ios --output-dir /tmp/check` (catches import/syntax errors), `npx expo-doctor`, and a simulator screenshot (`xcrun simctl io booted screenshot out.png`).
- **Architecture:** strict **two-layer design tokens** — `src/theme/primitives.js` (raw values transcribed 1:1 from the design screenshots) → `src/theme/theme.js` (semantic tokens: `colors.bg`, `colors.accent`, `typography.h1`, `spacing`, `radius`). Screens import only from the `src/theme` barrel and must use semantic tokens — **no hardcoded hex/sizes**. Brand accent is orange `#F05305`. Navigation shell in `src/navigation/RootNavigator.js` (native-stack over a **tiered tab bar**: the native system bar (`react-native-bottom-tabs`, in `NativeTabs.js`) on the modern tiers — Liquid Glass floating pill on iOS 26 (liquid-glass) and `BottomNavigationView` on Android 12+ (material-you) — falling back to JS `@react-navigation/bottom-tabs` everywhere else; the native module is lazy-`require`d so Expo Go never imports it. Tab icons go through `Icon`, which renders SF Symbols on iOS via the Expo-Go-bundled `expo-symbols`). One file per screen in `src/screens/` (some are still `ScreenPlaceholder` stubs). **Runtime UI tiers** — `useUITier()` + `TierProvider` swap only chrome (Card / ConfirmDialog / tab bar / rest bar) across liquid-glass/material-you/fallback; all per-tier styling is centralized in `src/theme/tiers.ts` + `TierSurface.tsx`, and the native glass/material deps lazy-load only on their tier so Expo Go stays on fallback. Mock data (`src/data/`) is deliberately isolated so **Supabase** swaps in behind stable shapes later — keep screen-facing shapes stable. Reuse shared components in `src/components/` rather than restyling per screen.
- **UI source of truth is screenshots, not Figma** — in `project/brief/screenshots/` (filenames contain a narrow no-break space before "PM"; glob, don't hardcode).

## Default Skill Routing

When skills overlap, follow these defaults unless the user explicitly asks for a different one:

- **Animation, motion, UI polish, transitions, micro-interactions (implementation/review)** → default to `emil-design-eng`. It has a decision framework (when *not* to animate), specific opinionated values with reasoning, and a required `Before | After | Why` review table format. Other skills (`interaction-design--animation-principles`, `interaction-design--micro-interaction-spec`) are only used when the user explicitly names them or asks for a generic principles primer / formal spec document.
- **Motion accessibility (vestibular safety, prefers-reduced-motion, photosensitivity)** → `inclusive-interaction--motion-sensitivity` always runs alongside Emil for any motion work — it's harm prevention, not taste, and is non-negotiable.
- **Code-level UI review (a11y, forms, hydration, dark mode, anti-patterns)** → `vercel-web-design-guidelines` for the file-by-file checklist pass.
- **React/Next.js performance** → `vercel-react-best-practices`. **React Native/Expo perf** → `vercel-react-native-skills`.
- **Bold aesthetic direction / building a new screen from scratch** → `frontend-design`.

## How Skills Layer Together
Designer skills build the system. Inclusive skills extend it for accessibility. Use together:
1. Design gestures → then design gesture alternatives
2. Build color system → then audit for color independence
3. Create personas → then expand with disability dimensions
4. Define animations → Emil for the motion taste, motion-sensitivity for the safety pass
5. Write component spec → then verify keyboard navigation
6. Create typography scale → then ensure flexible typography
7. Design layouts → then audit responsive accessibility

## Project Conventions
- Adapt all design outputs to the project's existing stack and conventions
- Prioritize accessibility (WCAG AA minimum) in all design decisions
- Use design tokens over raw values whenever possible
- Design for keyboard, screen reader, and reduced motion from the start
- Include disability as a natural dimension in all personas and user stories
